# 上传与大文件

AsterDrive 不要求你手动选择上传方式。
你只需要上传文件，系统会根据文件大小、当前工作空间和当前存储路线自动选择合适的处理方式。

## 普通用户先记住这几件事

- 小文件通常会很快完成
- 大文件通常会自动分片上传
- 上传中断后，系统会尽量提示继续未完成的上传
- 继续上传时，重新选择同一个文件即可

日常使用时，你不需要区分“普通上传”“分片上传”或“S3 直传”。

## 管理员要提前准备什么

如果你负责部署，真正需要提前想清楚的是:

- 当前用户或团队绑定到了哪个策略组
- 这个策略组里的规则会把文件分到哪条存储策略
- 这条策略的单文件大小上限和分片大小是否合理
- 服务器本地临时目录是否有足够空间
- 如果你用了反向代理，上传大小和超时是否足够
- 如果你用了 S3 / MinIO 直传，对象存储 CORS 是否已经配好

## 哪些上传会占用本地磁盘

并不是所有上传都会把整个文件先落到服务端本地磁盘。
最常见会明显占用本地临时目录的情况有:

- 使用默认本地存储，尤其是需要分片组装时
- 某些需要服务端参与处理的上传路径

所以管理员要提前确认服务器本地磁盘还有足够空间，尤其是:

- `data/.tmp`
- `data/.uploads`

## 如果你使用 S3 / MinIO，后台里最常见的两种方式

管理员在存储策略里最常见的两种 S3 上传方式是:

### 服务端流式中继

浏览器先把文件传给 AsterDrive，服务端直接中继到 S3。正常路径不落本地临时目录。

### Presigned 直传

浏览器直接把文件或分片上传到 S3 / MinIO。
这种方式通常最省服务端带宽，但对象存储必须先配好浏览器上传需要的 CORS。

## 上传失败时，先按这个顺序检查

1. 当前工作空间是不是切对了
2. 当前用户或团队绑定的是哪一个策略组
3. 该策略组里命中的存储策略单文件大小上限是否足够
4. 反向代理的请求体大小和超时时间是否足够
5. 如果走的是 S3 Presigned，对象存储的 CORS 是否已经配置正确
6. 用户或团队配额是否已经达到上限

## 什么时候应该改配置

如果用户经常上传大文件，管理员通常只需要关注这几处:

- `管理 -> 存储策略` 里的单文件大小上限和分片大小
- `管理 -> 策略组` 里的文件大小路由规则
- 服务器本地临时目录的可用空间
- 反向代理的上传大小和超时
- S3 / MinIO 的 CORS
