# 上传与大文件

AsterDrive 不要求用户手动选择上传模式。你只需要上传文件，系统会根据文件大小和当前存储策略自动选择合适的方式。

## 你会遇到的四种上传方式

| 模式 | 适合场景 | 数据流向 |
| --- | --- | --- |
| `direct` | 小文件、普通表单上传 | 浏览器 -> AsterDrive |
| `chunked` | 大文件、本地或普通代理上传 | 浏览器 -> AsterDrive（分片） |
| `presigned` | S3 小文件直传 | 浏览器 -> 对象存储 |
| `presigned_multipart` | S3 大文件分片直传 | 浏览器 -> 对象存储（multipart） |

## `direct`

这是最常见的小文件上传方式。

特点：

- 适合小文件
- 上传流程简单
- 上传流量经过 AsterDrive
- 同目录已有同名文件时，不会直接覆盖

## `chunked`

这是大文件上传的主要方式。

特点：

- 大文件会被拆成多个分片
- 失败时更容易重试
- 适合网络不稳定或上传时间很长的场景
- 上传流量仍然经过 AsterDrive 与反向代理

如果你的用户经常上传大文件，这一模式最依赖反向代理的超时和上传大小配置。

## `presigned`

这是 S3 或 MinIO 一类对象存储下的小文件直传方式。

特点：

- 浏览器直接把文件发到对象存储
- AsterDrive 只负责协调和收尾
- 可以明显减轻应用服务器和反向代理的上传压力

通常在文件不超过当前分片大小，或策略把分片大小设为 `0` 时使用。

## `presigned_multipart`

这是 S3 或 MinIO 一类对象存储下的大文件分片直传方式。

特点：

- 浏览器直接把每个分片上传到对象存储
- AsterDrive 负责创建 multipart upload、签发每个分片的上传地址，并在最后完成合并
- 适合大文件和不稳定网络
- 可以减少应用服务器和反向代理承载的大文件流量

只有在当前存储策略是 `s3`，开启了 Presigned 直传，并且文件超过当前分片大小时，系统才会使用这一模式。

## 管理员最需要关心什么

### 存储策略

上传方式不是全站统一配置，而是由当前命中的存储策略决定。管理员最常调的几个项目是：

- 存储驱动：本地或 S3
- 单文件大小上限
- 分片大小
- 是否启用 Presigned 直传

### 反向代理

- `direct` 和 `chunked` 会经过反向代理
- `presigned` 和 `presigned_multipart` 的主要上传流量不经过反向代理
- 所以经常上传大文件时，要先确认代理层没有限制请求体或超时

### S3 / MinIO 的 CORS

如果你启用了 Presigned 直传，对象存储必须允许浏览器直接上传，否则用户会在上传阶段直接失败。

## 上传异常时先查什么

上传异常时，优先检查：

1. 当前命中的存储策略是什么
2. 该策略的分片大小和 Presigned 直传是否符合预期
3. 代理层是否限制了上传体积或超时时间
4. 如果是 `presigned` 或 `presigned_multipart`，对象存储是否已经正确配置 CORS
5. 用户配额和单文件大小上限是否触发限制
