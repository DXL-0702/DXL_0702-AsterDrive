# 前端页面如何随服务一起发布

AsterDrive 不是“前后端完全分离”的部署方式。正常情况下，你只需要发布一个二进制或一个镜像，前端页面会跟着服务一起提供。

## 正常部署时会发生什么

发布构建时，前端 `frontend-panel/dist` 会被打进后端二进制。

这意味着：

- Docker 镜像、systemd 和单二进制都能直接用
- 页面、公开分享页和静态资源都由同一个 AsterDrive 服务返回
- 不需要额外再部署一套静态站点

## 什么时候会读取磁盘上的前端产物

运行时如果当前工作目录下存在：

```text
./frontend-panel/dist
```

后端会优先读取这个目录，而不是二进制里内嵌的资源。

这种方式更适合：

- 本地调试
- 临时替换前端页面
- 排查某个前端修复是否生效

但要注意，它会受到当前工作目录影响。

## 构建时如果没有前端产物会怎样

如果构建后端时 `frontend-panel/dist` 不存在：

- 后端仍能构建成功
- 但嵌入进去的是回退页，不是完整前端
- API 仍然可用，但 Web 页面不可正常使用

所以生产构建前，通常应该先执行：

```bash
cd frontend-panel
bun install
bun run build
cd ..
cargo build --release
```

## 部署时该怎么选

### 单二进制 / 容器发布

优先使用编译时嵌入。这是最推荐的方式，因为版本最容易保持一致。

### 本地调试或临时覆盖

可以使用运行时磁盘覆盖。这样做的好处是不必每次都重编译 Rust 二进制，但也更容易出现“后端已经升级、前端还是旧文件”的半更新状态。

## 对反向代理的影响

前端页面、公开分享页、静态资源和 API 都由同一个后端服务提供。

这意味着反向代理通常只需要把整个站点代理到 AsterDrive，不需要再拆出独立静态资源服务。

## 升级建议

升级时如果你走的是嵌入模式：

- 重新构建前端
- 再重新构建后端
- 发布新的单二进制或镜像

如果你走的是运行时覆盖模式：

- 除了升级后端，还要确认磁盘上的 `frontend-panel/dist` 是否也已经同步更新
- 避免“后端是新版本、前端还是旧 dist”的半更新状态
