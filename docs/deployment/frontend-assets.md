# 前端资源交付方式

AsterDrive 当前不是传统的“前后端完全分离部署”，而是以后端为主，前端资源可以嵌入到二进制里，也可以在运行时被磁盘上的 `dist` 覆盖。

## 两种来源

### 1. 编译时嵌入

正常发布构建时，前端 `frontend-panel/dist` 会被打进后端二进制。

特点：

- 单二进制部署
- 适合 Docker 镜像、systemd、裸机单文件运行
- 不需要单独再部署一套静态站点

### 2. 运行时磁盘覆盖

运行时如果当前工作目录下存在：

```text
./frontend-panel/dist
```

后端会优先读取这个目录，而不是二进制里内嵌的资源。

特点：

- 适合本地调试
- 适合快速替换前端产物
- 会受到当前工作目录影响

## 构建时如果没有前端产物会怎样

如果构建后端时 `frontend-panel/dist` 不存在：

- 后端仍能构建成功
- 但嵌入进去的是回退页，不是完整前端
- API 仍然可用，但 Web 页面不可正常使用

所以生产构建前，通常应该先执行：

```bash
cd frontend-panel
bun install
bun run build
cd ..
cargo build --release
```

## 部署时该怎么选

### 单二进制 / 容器发布

优先使用编译时嵌入。

优点：

- 产物简单
- 不需要再额外挂载前端静态文件
- 版本一致性最好

### 本地开发或临时覆盖

可以使用运行时磁盘覆盖。

优点：

- 不必每次都重编译 Rust 二进制
- 更适合快速验证文案和前端修复

但要注意：

- 工作目录错了就找不到覆盖目录
- 磁盘产物与嵌入版本不一致时，可能让排障变混乱

## 对反向代理的影响

前端页面、公开分享页、静态资源和 API 都由同一个后端服务提供。

这意味着反向代理通常只需要把整个站点代理到 AsterDrive，不需要再拆出独立静态资源服务。

## 升级建议

升级时如果你走的是嵌入模式：

- 重新构建前端
- 再重新构建后端
- 发布新的单二进制或镜像

如果你走的是运行时覆盖模式：

- 除了升级后端，还要确认磁盘上的 `frontend-panel/dist` 是否也已经同步更新
- 避免“后端是新版本、前端还是旧 dist”的半更新状态
