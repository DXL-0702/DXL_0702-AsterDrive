# 存储策略

先分清两个概念：

- `管理 -> 存储策略`：文件真正存到哪里
- `管理 -> 策略组`：用户或团队上传时，应该命中哪条存储策略

当前版本里，用户和团队都不是直接绑定单条存储策略，而是绑定一个策略组。  
策略组再按规则把上传分配到一条具体策略。

## 第一次启动后默认会有什么

新部署实例第一次启动后，系统会自动准备：

- 默认本地存储策略 `Local Default`
- 默认策略组 `Default Policy Group`

如果你什么都不改，新用户会自动绑定默认策略组，再由默认策略组把上传分到默认本地存储策略。

系统管理员创建新团队时，如果没有手动指定其他策略组，也会使用当前默认策略组。

## 当前支持哪两类存储策略

| 类型 | 说明 |
| --- | --- |
| `local` | 文件存到本地目录 |
| `s3` | 文件存到 S3 或兼容对象存储 |

## 什么时候只改存储策略，什么时候还要改策略组

- 只是想改“文件到底落本地还是 S3”这种真实存储目标：先看存储策略
- 想让不同用户、不同团队、不同大小的文件走不同路线：还要配策略组

最常见的后台操作顺序是：

1. 先创建或测试好存储策略
2. 再创建策略组规则
3. 最后把用户或团队绑定到目标策略组

## 存储策略里最常见的选项

| 项目 | 作用 |
| --- | --- |
| 名称 | 这条策略在后台里的显示名称 |
| 驱动类型 | `local` 或 `s3` |
| 连接信息 | 本地目录，或 S3 的 endpoint / bucket / 密钥 |
| 单文件大小上限 | 允许上传的最大文件大小；`0` 表示不限制 |
| 分片大小 | 大文件上传时每一片的大小 |
| 默认策略 | 新默认组或默认分流规则会优先使用它 |
| 附加选项 | 本地内容去重、S3 上传方式等 |

## 本地存储和 S3 怎么选

### 选 `local`

适合单机、NAS 和文件直接落本地磁盘的场景。

默认情况下，本地策略不会做内容去重。

如果你在后台开启“内容去重”：

- 上传完成后会再读一遍临时文件并计算内容指纹
- 相同内容会复用同一份底层文件，减少重复占用磁盘

如果保持关闭：

- 上传路径更直接，不会多一次全文件读取
- 相同内容也会分别保留成独立文件

### 选 `s3`

适合文件放到 MinIO、AWS S3 或其他兼容对象存储的场景。

## S3 上传方式怎么理解

### 服务端流式中继 `relay_stream`

浏览器先把文件传给 AsterDrive，服务端再直接中继到 S3。  
正常路径不落本地临时目录，也不会做内容去重。

### Presigned 直传 `presigned`

浏览器直接上传到 S3 / MinIO。  
文件不大于分片大小时会单次上传，更大的文件会自动走 S3 分片上传。

使用这种方式前，请确认对象存储已经配置好浏览器上传所需的 CORS，尤其是：

- 允许上传来源
- 允许 `PUT`
- `ExposeHeaders` 包含 `ETag`

## S3 下载方式怎么理解

### 服务端中继下载 `relay_stream`

AsterDrive 先从对象存储读取内容，再把字节流回传给浏览器。  
适合需要由应用节点继续控制响应头、同源下载行为或下游网络策略的场景。

### Presigned 重定向 `presigned`

AsterDrive 先做权限校验，再把浏览器重定向到一个短时效的 S3 `GET` URL。  
这样能把下载带宽和长连接压力交给对象存储，但最终响应头和缓存行为也会更多落到对象存储这一侧。

当前边界是：

- 登录态文件下载、团队文件下载、分享下载：会按存储策略的下载方式分流
- 公开直链 `/d/...` 和预览链路 `/pv/...`：仍然走服务端响应，不会改成重定向
- 分享下载次数：在生成可用下载响应后计数；命中 `304` 不会增加

使用 `presigned` 前，请确认：

- 客户端网络可以直接访问对象存储地址
- 对象存储允许返回期望的 `Content-Disposition` / `Content-Type`
- 你已经接受下载缓存行为更多由对象存储响应头决定

## 策略组怎么理解

策略组是一组有序规则。  
每条规则都会指定：

- 命中哪条存储策略
- 优先级
- 适用的文件大小范围

最简单的策略组只有一条规则：

- 任意大小的文件都走 `Local Default`

更常见的进阶做法是：

- 小文件走本地存储
- 大文件走 S3 / MinIO
- 某些团队绑定独立策略组，和个人空间分开走

## 使用时一定注意这几件事

- 使用本地存储时，目录最好写绝对路径
- 本地策略默认关闭内容去重；只有你明确开启后，系统才会额外计算内容指纹并复用同一份底层文件
- 使用 `presigned` 前，先确认对象存储已经配置好 CORS
- 使用下载 `presigned` 前，先确认客户端能直连对象存储，并理解带宽将从 AsterDrive 节点转移到对象存储
- 使用本地存储或少数需要服务端临时处理的上传路径时，服务器本地临时目录要有足够空间
- S3 的两种上传方式都不会做内容去重
- 单文件大小限制、策略组规则和用户或团队配额都会影响上传是否成功

## 哪些修改不要直接做

已经有文件写入的策略，不要直接修改这些决定实际存放位置的选项：

- 本地目录
- Bucket
- Endpoint

旧文件仍然会按原位置读取，直接改位置会导致已有文件找不到。

更稳妥的做法通常是：

1. 新建一条策略
2. 迁移旧数据
3. 把用户或团队切到使用新策略的策略组

## 日常维护建议

- 至少保留一条可用的默认存储策略
- 至少保留一个启用中的默认策略组
- 保存前先做一次连接测试
- 给不同用户或团队分配不同存储路线时，到 `管理 -> 用户` 或 `管理 -> 团队` 里绑定策略组
