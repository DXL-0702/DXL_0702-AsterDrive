# 存储策略

先分清两个概念，再去看字段名会更容易:

- `管理 -> 存储策略`: 文件真正存到哪里
- `管理 -> 策略组`: 用户或团队上传时，应该走哪条存储策略

当前版本里，用户和团队都不是直接绑定单条存储策略，而是绑定一个策略组。
策略组再按规则把上传分配到一条具体策略。

## 第一次启动后默认会有什么

新部署实例第一次启动后，系统会自动准备:

- 默认本地存储策略 `Local Default`
- 默认策略组 `Default Policy Group`

如果你什么都不改，新用户会自动绑定默认策略组，再由默认策略组把上传分到默认本地存储策略。

## 当前支持哪两类存储策略

| 类型 | 说明 |
| --- | --- |
| `local` | 文件存到本地目录 |
| `s3` | 文件存到 S3 或兼容对象存储 |

## 什么时候只改存储策略，什么时候还要改策略组

- 只是想改“文件放在本地盘还是 S3”这种真实存储目标: 先看存储策略
- 想让不同用户、不同团队、不同大小的文件走不同路线: 还要配策略组

最常见的后台操作顺序是:

1. 先创建或测试好存储策略
2. 再创建策略组规则
3. 最后把用户或团队绑定到目标策略组

## 存储策略页面里最常见的字段

| 字段 | 说明 |
| --- | --- |
| `name` | 策略名称 |
| `driver_type` | `local` 或 `s3` |
| `endpoint` | S3 服务地址；本地策略留空 |
| `bucket` | S3 存储桶名称 |
| `base_path` | 本地目录或对象前缀 |
| `max_file_size` | 单文件大小上限，单位字节；`0` 表示不限制 |
| `chunk_size` | 分片大小；后台按 MB 输入，`0` 表示尽量单次上传 |
| `is_default` | 是否为系统默认策略 |
| `options` | 用于附加策略开关，例如本地内容去重或 S3 上传方式 |

## 本地存储和 S3 怎么选

### 选 `local`

适合单机、NAS 和文件直接落本地磁盘的场景。

默认情况下，本地策略不会做内容去重。

如果你在后台开启 `内容去重`:

- 上传完成时会再读一遍临时文件并计算 SHA-256
- 相同内容会复用同一个 Blob，减少重复落盘

如果保持关闭:

- 上传路径更直接，不会多一次全文件读取
- 相同内容会各自落成独立 Blob

### 选 `s3`

适合文件放到 MinIO、AWS S3 或其他兼容对象存储的场景。

## S3 上传方式怎么理解

S3 策略当前有两种常见上传方式:

### `relay_stream`

浏览器先把文件传给 AsterDrive，服务端直接把字节流中继到 S3。正常路径不落本地临时目录。  
这种方式不做 SHA256 去重。

### `presigned`

浏览器直接上传到 S3 / MinIO。  
这种方式同样不做 SHA256 去重，而且对象存储必须先配好浏览器上传所需的 CORS。

## 策略组怎么理解

策略组是一组有序规则。
每条规则都会指定:

- 走哪条存储策略
- 优先级
- 适用的文件大小范围

最简单的策略组只有一条规则:

- 任意大小的文件都走 `Local Default`

更常见的进阶做法是:

- 小文件走本地存储
- 大文件走 S3 / MinIO
- 某些团队绑定独立策略组，和个人空间分开走

## 使用时一定注意这几件事

- 使用本地存储时，`base_path` 最好写绝对路径
- 本地策略默认关闭内容去重；只有你明确开启后，才会额外计算 SHA-256 复用 Blob
- 使用 `presigned` 前，先确认对象存储已经配置好 CORS
- 使用本地存储或少数需要服务端临时处理的上传路径时，服务器本地临时目录要有足够空间
- 使用任意 S3 上传方式时，服务端都不会做 Blob 去重
- 单文件大小限制、策略组规则和用户或团队配额都会影响上传是否成功

## 哪些修改不要直接做

已经有文件写入的策略，不要直接修改这些字段:

- `base_path`
- `bucket`
- `endpoint`

旧文件仍然会按原策略和原位置读取，直接改位置会导致已有文件找不到。

正确做法通常是:

1. 新建一条策略
2. 迁移旧数据
3. 把用户或团队切到使用新策略的策略组

## 日常维护建议

- 至少保留一条默认策略
- 至少保留一个可用的默认策略组
- 保存前先做一次连接测试
- 给不同用户或团队分配不同存储路线时，到 `管理 -> 用户` 或 `管理 -> 团队` 里绑定策略组
