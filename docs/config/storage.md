# 存储策略

存储策略在 `管理 -> 存储策略` 页面里维护，不在 `config.toml` 里。

它决定:

- 文件最终存到哪里
- 默认上传行为是什么
- 单文件大小限制是多少
- 新用户和指定用户默认走哪条策略

## 当前支持哪两类策略

| 类型 | 说明 |
| --- | --- |
| `local` | 文件存到本地目录 |
| `s3` | 文件存到 S3 或兼容对象存储 |

## 管理后台里最常见的字段

| 字段 | 说明 |
| --- | --- |
| `name` | 策略名称 |
| `driver_type` | `local` 或 `s3` |
| `endpoint` | S3 服务地址；本地策略留空 |
| `bucket` | S3 存储桶名称 |
| `base_path` | 本地目录或对象前缀 |
| `max_file_size` | 单文件大小上限，单位字节；`0` 表示不限制 |
| `chunk_size` | 分片大小；后台按 MB 输入，`0` 表示尽量单次上传 |
| `is_default` | 是否为系统默认策略 |
| `options` | 用于附加策略开关，例如本地内容去重或 S3 上传方式 |

## 第一次启动后默认会有什么

新部署实例第一次启动后，会自动创建一条默认本地策略:

- 名称: `Local Default`
- 驱动: `local`
- 路径: `data/uploads`
- 默认分片大小: `5 MiB`

新用户注册后，会自动分配当前系统默认策略。

## 本地存储和 S3 怎么选

### 选 `local`

适合单机、NAS 和文件直接落本地磁盘的场景。

默认情况下，本地策略不会做内容去重。

如果你在后台开启 `内容去重`:

- 上传完成时会再读一遍临时文件并计算 SHA-256
- 相同内容会复用同一个 Blob，减少重复落盘

如果保持关闭:

- 上传路径更直接，不会多一次全文件读取
- 相同内容会各自落成独立 Blob

### 选 `s3`

适合文件放到 MinIO、AWS S3 或其他兼容对象存储的场景。

## S3 上传方式怎么理解

S3 策略当前有三种常见上传方式:

### `proxy_tempfile`

浏览器先把文件传给 AsterDrive，服务端先写本地临时文件或临时分片，再写入 S3。兼容性最好，但会占用本地磁盘。  
这种方式也不做 SHA256 去重。

### `relay_stream`

浏览器先把文件传给 AsterDrive，服务端直接把字节流中继到 S3，不落本地临时分片目录。  
这种方式不做 SHA256 去重。

### `presigned`

浏览器直接上传到 S3 / MinIO。  
这种方式同样不做 SHA256 去重，而且对象存储必须先配好浏览器上传所需的 CORS。

## 使用时一定注意这几件事

- 使用本地存储时，`base_path` 最好写绝对路径
- 本地策略默认关闭内容去重；只有你明确开启后，才会额外计算 SHA-256 复用 Blob
- 使用 `presigned` 前，先确认对象存储已经配置好 CORS
- 使用 `proxy_tempfile` 或大文件上传时，服务器本地临时目录要有足够空间
- 使用任意 S3 上传方式时，服务端都不会做 Blob 去重
- 单文件大小限制和用户配额都会影响上传是否成功

## 哪些修改不要直接做

已经有文件写入的策略，不要直接修改这些字段:

- `base_path`
- `bucket`
- `endpoint`

旧文件仍然会按原策略和原位置读取，直接改位置会导致已有文件找不到。

正确做法通常是:

1. 新建一条策略
2. 迁移旧数据
3. 把用户默认策略切到新策略

## 日常维护建议

- 至少保留一条默认策略
- 保存前先做一次连接测试
- 给不同用户分配不同策略时，到 `管理 -> 用户` 里单独设置
